Decision: Supabase for all-in-one backend (PostgreSQL + Auth + Storage)

Database Schema

users (Supabase Auth)
  ↓ 1:many
quotes
  ├─ id (UUID, primary key)
  ├─ user_id (FK → auth.users)
  ├─ email (TEXT)
  ├─ project_description (TEXT)
  ├─ formats (TEXT[])
  ├─ status (ENUM: pending, in_progress, completed, cancelled)
  └─ created_at (TIMESTAMP)
  ↓ 1:many
quote_files
  ├─ id (UUID, primary key)
  ├─ quote_id (FK → quotes)
  ├─ file_name (TEXT)
  ├─ storage_path (TEXT)
  └─ is_result (BOOLEAN)  -- Distinguishes uploads from completed vectors

Key Design Decisions:

  • Row Level Security (RLS) for multi-tenant data isolation
  • Cascade deletes for GDPR compliance
  • Normalized schema (3NF) for data integrity

Key Product Decisions

Decision 1: Hybrid Guest/Authenticated Model

Context: Should users create accounts before submitting quotes?

Options:

  • A) Require login → Higher friction, better retention
  • B) Guest-only → Lower friction, no history
  • C) Hybrid → Allow both

Decision: Hybrid approach

Rationale:

  • First-time users have low commitment (friction kills conversion)
  • Repeat customers see value after first quote (organic incentive to register)
  • Data retention happens either way (guest submissions stored)

Success Metrics:

  • Guest-to-registered conversion: Target >20%
  • Dashboard return rate: Target >40%

Decision 2: Database-First Architecture

Context: Where should quotes be stored as source of truth?

Decision: Database primary, email as notification fallback

Database-First Benefits:

  • Reliability: Guaranteed persistence vs. email delivery uncertainty
  • Scalability: No attachment size limits
  • User Experience: Always-available dashboard
  • Operations: Centralized quote management

Trade-offs:

  • More complex architecture
  • Dependency on Supabase uptime
  • BUT: Better long-term foundation for growth

Technical Challenges & Solutions

Challenge 1: Fake Vectorization Feature Pivot

Issue: Attempted to add automated preview using Potrace library

Technical Barrier:

  • Native C dependencies incompatible with Vercel serverless
  • Workarounds produced low-quality geometric patterns (not real vectorization)

Critical Decision Point:

  • Option A: Spend more time debugging → Sunk cost fallacy
  • Option B: Ship "fake" feature → Damages trust
  • Option C: Cut feature entirely → Authentic positioning

Decision: Removed feature completely

Impact:

  • Clearer value proposition (manual = quality)
  • Faster deployment (no technical debt)
  • Better brand trust

Challenge 2: Multi-Select Component UX

Issue: Shadcn multi-select had command palette search (overkill for 6 options)

User Feedback: "Doesn't work, doesn't need search"

Solution: Simplified to checkbox dropdown with horizontal badge display

Approach: User feedback > framework defaults

Challenge 3: useSearchParams Build Error

Issue: Next.js pre-rendering failing on login page

Root Cause: useSearchParams requires Suspense boundary for SSR

Solution: 

<Suspense fallback={<div>Loading...</div>}>
  <LoginForm />
</Suspense>

Learning: Framework requirements for SSR/SSG compatibility aren't optional
